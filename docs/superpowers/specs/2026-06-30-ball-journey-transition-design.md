# ボールジャーニー演出（3スポーツ・スクロール連動トランジション）設計書

## 背景・目的

OHZI Interactive（ohzi.io）のスクロール連動シーン遷移・ワープ演出に着想を得つつ、特定企業のクリエイティブ（宇宙ワープ・ネオンポータル・地形モーフィング等）をそのまま模倣するのではなく、yusuさんの「体育教師→エンジニア」という自分のストーリーに翻訳する。

具体的には、既存のクリスタル（発光する球体）を「ボール」に見立て、サッカー→バスケットボール→バレーボールという3つの球技を、セクション間の接続演出として使う。OHZIサイトで印象的だった「前に進んでいく」「水中を潜っていく」ようなカメラの没入感を、ドリブル・パス・シュート・スパイクといった球技の動きで再現する。

## スコープ

### 含むもの
- **① サッカーシーン**（Hero → Story間）: ドリブルで前進 → ロングパス
- **② バスケットボールシーン**（Story → Projects間）: キャッチ → パス回し → シュート → リングイン
- **③ バレーボールシーン**（Projects → Contact間）: レシーブ → トス → スパイク
- 各シーンは既存のCrystal（球体）をボールとして再利用
- 人物パーツ（手・上半身のみ）は無料素材＋トーンシェーダーでサイトの世界観に統一
- GSAP ScrollTrigger（scrub）によるスクロール連動制御
- `prefers-reduced-motion` 対応

### 含まないもの（将来検討・今回見送り）
- 銀河・惑星ナビゲーション（VIVERSE風）
- 宇宙飛行士などのフルボディキャラクター
- ステンシルマスクによるポータル型プロジェクトモーダル
- 複雑なフルボディ人体アニメーション（手・上半身の部分アセットで代替）

既存のセクション構成（Hero / Impact / Story / Projects / Skills / Blog / Contact）とその中身（タイムライン文言・PSIカード等）は変更しない。3スポーツ演出はあくまで**セクション間の「つなぎ」**として機能させる。

## アーキテクチャ概要

### 新規ディレクトリ構成
```
src/components/canvas/journey/
  ├─ BallJourney.tsx      # 3シーンを統括するコンテナ。スクロール進捗で表示切替
  ├─ SoccerScene.tsx       # ①ドリブル+ロングパス
  ├─ BasketballScene.tsx   # ②キャッチ+パス回し+シュート
  ├─ VolleyballScene.tsx   # ③レシーブ+トス+スパイク
  ├─ CourtSurface.tsx      # 共通のコート地面（PBRテクスチャ+ノイズ変形、詳細は後述）
  ├─ GrassField.tsx        # サッカーシーン専用：風になびくインスタンス芝
  ├─ Floodlights.tsx       # ナイター照明（投光器ジオメトリ+強い点光源+GodRays用メッシュ）
  └─ useHumanAsset.ts      # 手モデルのロード＋トーンシェーダー適用を共通化
```

### データフロー
- 既存の `CameraRig.tsx` と同様、ページ全体のスクロール進捗（0〜1）を取得する仕組みを使う
- 各シーンは自分の担当区間（例: サッカー = 0.05〜0.18）でのみ `visible: true` にし、区間内progressでボール位置・カメラ位置を補間
- ボール軌跡: ドリブル/パスは `CatmullRomCurve3`、シュート/スパイクは手計算の放物線（`y = v*t - 0.5*g*t²`）

### カメラ移動モデル（重要・明確化）
スクロール＝画面が縦に流れるだけ、ではない。**カメラはボールの軌跡（`CatmullRomCurve3`）に沿って3D空間内を前進・旋回しながら移動する**「奥へ進んでいくドリー」モデルを採用する。スクロール量はあくまで「曲線上のどこまで進んだか」という1つのprogress値に変換され、カメラ位置・注視点・FOVをその値で都度計算し直す（[Codropsのscroll-drivenカメラパス手法](https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/)と同様のアプローチ）。実際に縦スクロールしているのはDOM上のダミー高さ要素であり、見た目上は常に「前進する一人称/三人称ショット」になる。

## 各シーン詳細

### ① サッカーシーン（Hero → Story）
- 既存の `explore-click` イベント＋スクロール進捗をトリガーに使う
- ボール（クリスタル）が地面でバウンドしながら前進（サインカーブの縦バウンド＋`CatmullRomCurve3`の横移動）
- `@react-three/drei` の `<Trail>` でオレンジの軌跡
- 地面は実写PBRテクスチャ（芝）＋風になびくインスタンス芝（`GrassField.tsx`、詳細は次節）。センターサークル・サイドラインはテクスチャに焼き込むか、薄い発光ラインメッシュを重ねる
- 終盤、ロングパスとして放物線を描いて画面奥へ。蹴る人物は描画せず、ボール自体の急加速・回転で表現（人物アセット不要）
- カメラはボールを追従するロー・アングルのドリーショット（前項「カメラ移動モデル」参照）
- 雰囲気はナイター（夜の競技場、投光器照明）。詳細は次節「背景・環境演出」

### ② バスケットボールシーン（Story → Projects）
- 飛んできたボールを、画面奥から伸びる「手」（無料低ポリ手モデル＋トーンシェーダー）がキャッチ
- 2〜3個の手モデルの間でパス（放物線移動）
- シュート軌道（放物線）でリングへ。リング＝Torus、ネットは簡易ライン
- リング通過時にBloom強化＋パーティクルバーストで「スコア」演出
- 着地と同時にProjectsセクション開始に同期

### ③ バレーボールシーン（Projects → Contact）
- 落下するボールを画面下から伸びる「手」がレシーブ
- レシーブ → トス（ほぼ垂直の山なり放物線）→ スパイク（急角度で打ち込む）
- スパイクのインパクトでフラッシュ＋軽いカメラシェイク → Contactセクションへ着地

## 背景・環境演出（リアリティ強化）

色付きのシェーダー平面だけでは「チープな画」になるため、以下3点セットで質感を作る。山岳地形は今回不要と判断し、**ナイター（夜の競技場・投光器照明）**を世界観の軸に据える。既存のダークテーマ・クリスタルの発光表現とも相性が良い。

### ① 風になびく芝（サッカーシーンのみ）
- インスタンシング＋GLSL頂点シェーダーで、芝ブレード1本ずつをサインカーブ＋ノイズで揺らす（[実装例](https://github.com/Nitash-Biswas/grass-shader-glsl)、[Codrops解説](https://tympanus.net/codrops/2025/02/04/how-to-make-the-fluffiest-grass-with-three-js/)）。購入素材不要、コードのみで実現
- パフォーマンス上限を決めて（例: 数万本オーダー）モバイルでは本数を落とす、または静的テクスチャにフォールバック

### ② 実写PBRテクスチャ（コート面全般）
- [Poly Haven](https://polyhaven.com/textures) / [ambientCG](https://ambientcg.com/) のCC0テクスチャ（クレジット表記不要）を使用
- サッカー: 芝の地面テクスチャ／バスケ: 体育館の木目フロア／バレー: 砂 or 体育館床
- カラー・法線・ラフネスマップを `meshStandardMaterial` に設定し、フラットな単色シェーダーより質感を底上げする
- `public/textures/` に配置し、ライセンスはCC0のみ採用（個別ファイルのライセンス確認は実装時に必須）

### ③ ナイター照明（God Rays）
- 投光器: シンプルな円柱（ポール）＋発光ディスク（ライトフィクスチャ）のジオメトリを各シーンの四隅程度に配置。複雑なモデル不要
- 強い `pointLight` / `spotLight` を投光器位置に設置し、Bloomで滲ませる
- `@react-three/postprocessing`（**既存の依存関係、追加インストール不要**）の `GodRays` エフェクトで投光器から伸びる光条を表現（[公式ドキュメント](https://react-postprocessing.docs.pmnd.rs/effects/god-rays)）
- 空は既存のダークカラー（`#0a0a0f`系）を基調にした夜空。地平線近くにわずかな色温度の違い（投光器の暖色グロー）を持たせる程度に留め、山などの大規模背景要素は作らない
- 観客・スタンドのにぎわいは、未使用の [`Particles.tsx`](../../../src/components/canvas/Particles.tsx) を遠景・小スケール・低速・暖色寄りに調整して再利用し、「ボケた照明・人の気配」を安価に表現する（実在の観客モデルは配置しない）

### 適用範囲
- ①②③はサッカーシーンを基準に設計し、バスケ（屋内体育館）・バレー（屋内 or ビーチ）はライト色温度とテクスチャ種類を変える形で同じ仕組みを使い回す
- 屋内シーン（バスケ）は投光器の代わりに体育館の天井照明（同じくpointLight＋GodRays転用可）を想定

## アセット調達方針

- **手モデル**: Sketchfabの **CC0ライセンス**低ポリ手モデルのみを採用（CC-BY等、クレジット表記が必要なライセンスは除外し、表記不要なCC0に限定する）
- **テクスチャ**: Poly Haven / ambientCGの**CC0**PBRテクスチャ（芝・木目フロア・砂）。4K原寸は重いため、用途に応じて1K〜2Kにリサイズして使用する
- ダウンロードした `.glb` は `public/models/`、テクスチャは `public/textures/` に配置
- マテリアル: `MeshToonMaterial`（スキニング対応・自作シェーダーの互換性問題を回避）＋既存の `Effects.tsx` のBloomで発光統一。色は既存ブランドカラー（`--color-or` `--color-am`）に寄せる
- 採用するモデルは静的メッシュ（非スキニング）の手のみを基本とし、複雑な人体アニメーションは扱わない

## パフォーマンス・アクセシビリティ

- `prefers-reduced-motion` 時は全シーンをスキップし、現行のシンプルな `scrollIntoView` 遷移にフォールバック（[global.css](../../../src/styles/global.css) の既存規約に準拠）
- モバイルでは手モデル・芝インスタンス数・GodRaysを段階的に間引く（テクスチャ解像度を落とす／芝の本数を減らす／GodRaysを通常のBloomグローのみに簡略化）
- PBRテクスチャはリサイズ・圧縮（WebP/KTX2等）してアセットサイズを抑える
- 実装後、Phase 3-9のLighthouse計測で悪化がないか確認する

## オープンクエスチョン・リスク

- Sketchfabの個別モデルの正確なライセンス・ポリゴン数は、実際にダウンロードして検証する必要がある
- Poly Haven / ambientCGのテクスチャも個別ファイルのライセンス（基本CC0想定）を実装時に確認する
- インスタンス芝・GodRaysはGPU負荷が高い演出のため、モバイル・低スペック端末での実機検証が必須（Phase 3-9で既に課題になっているdpr設定と合わせて検討）
- 実装規模が大きいため、①②③を別々のIssue/PRに分割し、段階的に進める（既存のIssue→branch→PRワークフローに準拠）
- 3シーンを1つのスクロール体験として違和感なく繋げる調整には、実装後の試行錯誤が必要になる見込み
