# ボールジャーニー演出（3スポーツ・スクロール連動トランジション）設計書

## 背景・目的

OHZI Interactive（ohzi.io）のスクロール連動シーン遷移・ワープ演出に着想を得つつ、特定企業のクリエイティブ（宇宙ワープ・ネオンポータル・地形モーフィング等）をそのまま模倣するのではなく、yusuさんの「体育教師→エンジニア」という自分のストーリーに翻訳する。

具体的には、既存のクリスタル（発光する球体）を「ボール」に見立て、サッカー→バスケットボール→バレーボールという3つの球技を、セクション間の接続演出として使う。OHZIサイトで印象的だった「前に進んでいく」「水中を潜っていく」ようなカメラの没入感を、ドリブル・パス・シュート・スパイクといった球技の動きで再現する。

## スコープ

### 含むもの
- **① サッカーシーン**（Hero → Story間）: ドリブルで前進 → ロングパス
- **② バスケットボールシーン**（Story → Projects間）: キャッチ → パス回し → シュート → リングイン
- **③ バレーボールシーン**（Projects → Contact間）: レシーブ → トス → スパイク
- 各シーンは既存のCrystal（球体）をボールとして再利用
- 人物パーツ（手・上半身のみ）は無料素材＋トーンシェーダーでサイトの世界観に統一
- GSAP ScrollTrigger（scrub）によるスクロール連動制御
- `prefers-reduced-motion` 対応

### 含まないもの（将来検討・今回見送り）
- 銀河・惑星ナビゲーション（VIVERSE風）
- 宇宙飛行士などのフルボディキャラクター
- ステンシルマスクによるポータル型プロジェクトモーダル
- 複雑なフルボディ人体アニメーション（手・上半身の部分アセットで代替）

既存のセクション構成（Hero / Impact / Story / Projects / Skills / Blog / Contact）とその中身（タイムライン文言・PSIカード等）は変更しない。3スポーツ演出はあくまで**セクション間の「つなぎ」**として機能させる。

## アーキテクチャ概要

### 新規ディレクトリ構成
```
src/components/canvas/journey/
  ├─ BallJourney.tsx      # 3シーンを統括するコンテナ。スクロール進捗で表示切替
  ├─ SoccerScene.tsx       # ①ドリブル+ロングパス
  ├─ BasketballScene.tsx   # ②キャッチ+パス回し+シュート
  ├─ VolleyballScene.tsx   # ③レシーブ+トス+スパイク
  ├─ CourtSurface.tsx      # 共通のコート地面シェーダー（GroundRippleの手法を再利用）
  └─ useHumanAsset.ts      # 手モデルのロード＋トーンシェーダー適用を共通化
```

### データフロー
- 既存の `CameraRig.tsx` と同様、ページ全体のスクロール進捗（0〜1）を取得する仕組みを使う
- 各シーンは自分の担当区間（例: サッカー = 0.05〜0.18）でのみ `visible: true` にし、区間内progressでボール位置・カメラ位置を補間
- ボール軌跡: ドリブル/パスは `CatmullRomCurve3`、シュート/スパイクは手計算の放物線（`y = v*t - 0.5*g*t²`）

## 各シーン詳細

### ① サッカーシーン（Hero → Story）
- 既存の `explore-click` イベント＋スクロール進捗をトリガーに使う
- ボール（クリスタル）が地面でバウンドしながら前進（サインカーブの縦バウンド＋`CatmullRomCurve3`の横移動）
- `@react-three/drei` の `<Trail>` でオレンジの軌跡
- 地面はピッチシェーダー（緑、白線でセンターサークル・サイドライン）。`Scene.tsx` の `GroundRipple` と同じUVベースのシェーダー手法を流用
- 終盤、ロングパスとして放物線を描いて画面奥へ。蹴る人物は描画せず、ボール自体の急加速・回転で表現（人物アセット不要）
- カメラはボールを追従するロー・アングルのドリーショット

### ② バスケットボールシーン（Story → Projects）
- 飛んできたボールを、画面奥から伸びる「手」（無料低ポリ手モデル＋トーンシェーダー）がキャッチ
- 2〜3個の手モデルの間でパス（放物線移動）
- シュート軌道（放物線）でリングへ。リング＝Torus、ネットは簡易ライン
- リング通過時にBloom強化＋パーティクルバーストで「スコア」演出
- 着地と同時にProjectsセクション開始に同期

### ③ バレーボールシーン（Projects → Contact）
- 落下するボールを画面下から伸びる「手」がレシーブ
- レシーブ → トス（ほぼ垂直の山なり放物線）→ スパイク（急角度で打ち込む）
- スパイクのインパクトでフラッシュ＋軽いカメラシェイク → Contactセクションへ着地

## アセット調達方針

- **手モデル**: Sketchfabの **CC0ライセンス**低ポリ手モデルのみを採用（CC-BY等、クレジット表記が必要なライセンスは除外し、表記不要なCC0に限定する）
- ダウンロードした `.glb` は `public/models/` に配置
- マテリアル: `MeshToonMaterial`（スキニング対応・自作シェーダーの互換性問題を回避）＋既存の `Effects.tsx` のBloomで発光統一。色は既存ブランドカラー（`--color-or` `--color-am`）に寄せる
- 採用するモデルは静的メッシュ（非スキニング）の手のみを基本とし、複雑な人体アニメーションは扱わない

## パフォーマンス・アクセシビリティ

- `prefers-reduced-motion` 時は全シーンをスキップし、現行のシンプルな `scrollIntoView` 遷移にフォールバック（[global.css](../../../src/styles/global.css) の既存規約に準拠）
- モバイルでは手モデルを省略し、ボールの軌道のみ表示する簡略版を検討
- 実装後、Phase 3-9のLighthouse計測で悪化がないか確認する

## オープンクエスチョン・リスク

- Sketchfabの個別モデルの正確なライセンス・ポリゴン数は、実際にダウンロードして検証する必要がある
- 実装規模が大きいため、①②③を別々のIssue/PRに分割し、段階的に進める（既存のIssue→branch→PRワークフローに準拠）
- 3シーンを1つのスクロール体験として違和感なく繋げる調整には、実装後の試行錯誤が必要になる見込み

## 実装順序（提案）

1. 共通基盤: `BallJourney.tsx` / `CourtSurface.tsx` / スクロール進捗連携の土台
2. サッカーシーン実装・検証（人物アセット不要なので最も着手しやすい）
3. バスケットボールシーン実装・検証（手アセット調達を含む）
4. バレーボールシーン実装・検証
5. 統合テスト・パフォーマンス計測
