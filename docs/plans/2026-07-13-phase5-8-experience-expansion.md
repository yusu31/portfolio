# 3Dスクロールジャーニー体験拡張計画 — Phase 5〜8 再編

**日付:** 2026-07-13
**状態:** ユーザー承認済み(実装はPhase 5-1から)
**関連:** Issue #236 / `docs/HANDOFF_PHASE3-27.md`

## Context

Phase 4完了後のユーザーレビューで3つの本質的な指摘を受けた:

1. **旅が短い** — 参考サイト(Sébastien Lempens)を実測した結果、3チャプター・総ホイール入力約20万px・連続スクロール約80秒。対して現状は7ページ・約5,000px・十数秒。体験量として**約3〜4倍**が目標(ユーザー回答「パリのサイトに合わせたい」)。
2. **クリスタル球を旅の主人公に** — 参考サイトの「スクーターに乗る人物」(実装上も `motorbike` クラスを確認)に相当する存在として、クリスタル球が全区間を移動する。ユーザー指定のストーリーボード(一言一句尊重):
   **「サッカーでドリブル→ロングパス→バスケでキャッチ→フリースローシュート→リングを通過→そのまま下へ落ちてバレーコートへ→レシーブ→トス→アタック→最後はContactページへ」**
3. **チープさ脱却(アセット多め)** — 現ヴェニューは単色プリミティブのみ(テクスチャ0・GLTF 0・接地影なし)。CC0/CC-BYフリーアセット+HDRI+影+テクスチャで質感を引き上げる(ユーザー選択:「アセット多め」+「場合によってはモデリングや3Dサイトからの無料DLも考えて」)。
4. **終端演出の刷新** — 現在の「表彰台+3ボール」はかっこ悪いので廃止(ユーザー判断)。代案4つから**「旅路のミニチュアジオラマ」**を選択: 丸い石のテーブルに旅の全コース(3ヴェニュー+道)のミニチュアが載っていて、リレーを終えたクリスタル球がその終着点に転がり込んで静止する。「ポートフォリオ全体を一望する」入れ子構造の締め。

これに伴い旧Phase 5(公開準備)・旧Phase 6(公開前チェック)は**Phase 7・8へスライド**する。

## Phase 5: 経路延長 + ボールリレー(旅の主人公化)

### 5-1. 弧長パラメータ化への移行(単独PR・先行)

`CatmullRomCurve3.getPoint(t)`(制御点数依存)から **`getPointAt(u)`(弧長=距離割合)** へ移行し、`curveType` を `'centripetal'` に変更。以後「点を足すと全区間ズレる」問題が構造的に解消され、経路増設に強くなる。

- `src/journey/CameraRig.tsx` の `getPoint` 2箇所を `getPointAt` に置換
- `src/journey/sections.ts` を `src/journey/path/` に分割。`arcLengthDivisions` を明示し弧長キャッシュを温める
- 終端の「静止」は制御点をほぼ重ねる現行の力技をやめ、`CameraRig` 側で offset を `min(offset, 0.985)` 程度にクランプして表現(弧長化とゼロ長セグメントは相性が悪い)
- **注意**: パラメータ化と curveType の変更で「同じoffset→同じ位置」は保たれない。SECTION_RANGES は各ヴェニューが現行と同等のフレーミングになるよう再導出する(ピクセル一致ではなく「構図の同等性」をQA基準にする)
- Vitest回帰テスト追加(`curve.getLength()` レンジ、SECTION_RANGES 単調性、各uでのz座標がヴェニュー近傍にあること)

### 5-2. 経路延長(66→約200ユニット、PAGES 7→21)

- ヴェニュー間に3つのtransit区間の骨格を追加(この時点ではプレースホルダー地面のみ)
- fog far値の再検証(46→60〜70)、damping 0.25維持で実機比較
- QA: offsetサンプルを20〜24点に増やしてフルスイープ→Sonnet 2巡(間延び感・破綻の有無)
- **完了時にユーザーの通しスクロール確認を挟み、「旅の長さの体感」をここで承認してもらってから先へ進む**(3倍で足りなければ4倍へ調整)

### 5-3 / 5-4. ボールリレー実装(前半: サッカー→バスケ / 後半: バレー→Contact)

新モジュール `src/journey/ball/` を新設。**ボール位置は offset を入力とする純粋関数**(カメラ・カード表示と同じ「offsetが唯一の真実」原則):

- `anchors.ts` — リング中心・キャッチ点・レシーブ点などビート境界の共有座標(venues.tsxと単一ソース共有で継ぎ目ズレを構造的に防ぐ)
- `beats.ts` + `beats/*.ts` — ストーリーボードをビート列にマップ: `idle → dribble(バウンド弧の連結+左右ウィーブ) → pass(大放物線) → catch → freeThrow(終端=リング中心座標で「正確にくぐる」を保証) → fall(自由落下) → receive → setToss → spike → rest(Contact着地)`
- `ballPath.ts` — `getBallPose(u) → { position, focusWeight }`。CameraRigは `LOOKAT_PATH.getPointAt(u)` とボール位置を focusWeight(見せ場0.6〜0.85、上限<1.0でPhase 4の太陽回避を温存)でブレンド
- 演者はクリスタル球そのもの(CrystalBall.tsx にoffset連動positionを追加。材質不変)。サッカー/バスケ/バレーの静的ボールと**Contact表彰台+3ボールは撤去**
- 終端 `rest` ビート: クリスタル球はプラザ中央のテーブル(この時点ではプレースホルダーの円柱台座)に転がり込んで静止する。ジオラマの見た目本体はPhase 6-5で実装
- テストで担保: ①ビート継ぎ目の距離閾値(瞬間移動検知) ②全uサンプルでボールのNDC座標がフレーム内(|x|<0.9, |y|<0.85)に収まること(ブラウザ不要の一次防衛線)
- 物理エンジンは導入しない(振付ベース。演出の確実性とバンドル予算のため)

## Phase 6: アセットリッチ化(3d-assetsスキルの規約に従う)

**アセットごとに4ルートから最適を選ぶ**(ユーザー指示「場合によってはモデリングや3Dサイトからの無料DLも考えて」):
1. **無料素材庫DL**(Poly Haven / Sketchfab / Quaternius等、CC0/CC-BY) — 汎用物はまずここ
2. **プロシージャル**(R3Fコードで生成) — 既存ヴェニューの流儀。反復物・抽象物に
3. **Blenderモデリング**(bpyスクリプト生成→必要ならライトベイク→GLB出力) — 素材庫に無い固有物(観客席・ジオラマのテーブル等)。Lempens本人のパイプラインと同型
4. **AI生成**(無料枠・実行前にユーザー承認必須) — 上記で入手不能な唯一無二の物のみ

Splineは既却下(バンドル2MB超)のまま維持。**着手時にまず候補表(ルート・ライセンス・サイズ・工数)をユーザーに提示して承認を得る**(ライセンス義務・無料枠消費があるため勝手に進めない)。調査済み候補:

| 素材 | 出典 | ライセンス | 用途 |
|---|---|---|---|
| stadium_01 HDRI (1k) | Poly Haven | CC0 | 環境光/反射のみ(`background=false`、可視の空は既存Skyを維持) |
| サッカーゴール (21k tri) | Sketchfab/TepidGames | CC-BY(要表記) | Projects |
| バスケフープ (2.1k tri) | Sketchfab | CC-BY(要表記) | Skills(リング実座標を `ball/anchors.ts` に反映) |
| バレーネット | Sketchfab | CC-BY(要表記) | About |
| 木・フェンス | Quaternius / Sketchfab | CC0中心・個別確認 | Transit 1並木(drei `<Instances>`) |
| 観客席 | 未確認 → 見つからなければルート3(Blender)かルート2(プロシージャル) | — | Transit 2 |
| ジオラマのテーブル | ルート3(Blender)かルート2 | — | 終端見せ場(6-5) |

PRスライス:
1. **6-1 パイプライン基盤**: gltfjsx導入(`--transform`=Draco+WebP圧縮)、デコーダ自前配置(`public/draco/`、CDN依存排除)、`public/models/` 規約、`public/CREDITS.md` + フッター帰属表示UI
2. **6-2 HDRI導入(単独PR)**: **最大のリスク=クリスタル球はコード不変でも環境マップで見た目が変わる**。球のoffset固定前後比較スクリーンショットをマージ必須ゲートにする
3. **6-3 ヴェニューGLTF化**: ゴール/フープ/ネット差し替え+未配線の芝テクスチャ(`public/textures/leafy_grass_*`)配線+ContactShadows(ヴェニュー単位・小規模)。**フォトリアルPBRテクスチャは剥がして既存パレットの単色に置換**(taste保護の必須工程)
4. **6-4 transit風景**: 並木(Instances)・観客席・柵。Transit 3はGLB追加なしで既存`<Cloud>`の雲間演出(予算温存+3区間の差別化)
5. **6-5 旅路のミニチュアジオラマ(終端見せ場)**: 丸い石のテーブル+全コースのミニチュア。**ミニチュアは既存ヴェニューコンポーネントの縮小再利用**(scale groupで実体を共有→統一感が構造的に保証され、6-3のGLTF化が自動で反映される)+経路のリボン(TubeGeometry)。クリスタル球がミニチュアの終着点位置に静止する入れ子の締め。追加draw callはInstances/低解像度化で抑制
6. **6-6 Text3D押し出しタイトル**: まずPROJECTSの1つだけスパイクして可読性QA→OKなら全展開。**白ワープ/液体インク転換は見送り**(「1つの連続空間」という差別化と矛盾。決め瞬間の短いフラッシュ案のみオプションとして残し、実装前にtaste承認必須)

完了時: 採用/却下ルートを Obsidian `Decisions/` に記録(3d-assetsスキル⑥)。

## 実行ロジスティクス

- **最初のPR**: 新ロードマップをHANDOFFにdocs PRとして反映(Phase番号のスライド読み替えを明記)+ Obsidian記録
- モデル運用: 機械的実装・QA事務はSonnet、ボール軌道の演出調整・構図判断はFable
- 各PRの検証: tsc/ビルド → Vitest(経路・ボールのテスト) → Playwright offset直指定スクリーンショット → Sonnetサブエージェント2巡 → マージ
- Phase 5-2完了時とPhase 6完了時に**ユーザーの通しスクロール確認**を必須マイルストーンとする

## 検証方法(全体)

1. `pnpm build` + Vitest が全PR緑
2. devサーバーで offset 0→1 フルスイープのスクリーンショット(20点以上)をSonnet QAに委任
3. ボールリレーはNDCフレーム内テスト+継ぎ目テストで数値担保後、目視QA
4. Phase 6はアセット導入前後の比較スクリーンショット(特にクリスタル球)
5. 最終的にユーザー本人の通しスクロールで「旅の長さ」「球の物語」「質感」の3点を承認
