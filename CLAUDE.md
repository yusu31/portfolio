## 開発原則

- **Issue → branch → PR → merge** — コード・ドキュメントを問わず全変更でこの順を必ず守る（例外なし。HANDOFF文書も含む）
- コーディング前に関連ドキュメントを確認（全部読まない）
- 実装前に提案・確認してから着手

## GitHub ワークフロー

- Issue: https://github.com/yusu31/portfolio/issues
- ブランチ: `feature/{説明}-#{Issue番号}` 形式
- main への直接プッシュ禁止（ドキュメントのみの変更も例外なくPRを通す）
- **マージは自動化済み**: PRがマージ可能な状態になったら、Claudeはユーザーに確認を取らずそのまま `gh pr merge` を実行してよい（このプロジェクトに限り事前承認済み）。マージ後のローカルbranch削除・`git remote prune origin` も続けて自動実行する
